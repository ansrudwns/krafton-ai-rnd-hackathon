# [KRAFTON AI 해커톤 회고] 수학적으로 완벽했던 모델이 실전에서 무너진 이유: 시계열 검증(Validation)의 함정

이번 KRAFTON AI R&D Hackathon 결선에서 배틀로얄 매치의 킬 수를 예측하는 시계열 예측 모델을 개발했습니다. 코어 알고리즘인 **Bradley-Terry 모델과 마코프 체인(Markov Chain) 기반의 확률 분포 연산**은 우아하고 수학적으로 완벽했지만, 아쉽게도 최종 예측 점수에서 큰 오차를 기록하며 예상치 못한 결과를 얻었습니다. 

대회가 끝난 후, 코드를 한 줄씩 뜯어보며 뼈아픈 사후 분석(Post-mortem)을 진행했습니다. 본 포스팅에서는 **알고리즘이 완벽했음에도 불구하고 почему(왜) 데이터 사이언스의 함정에 빠졌는지, 그리고 이를 어떻게 수정했는지** 공유하고자 합니다.

---

## 1. 완벽했던 핵심 로직 (What Went Right)

해커톤의 목표는 총 10명의 플레이어가 50일에 걸쳐 진행하는 배틀로얄 경기 중, 마지막 29일(Day 22~50) 동안의 각 플레이어 총 킬 수를 예측하는 것이었습니다. 

이를 풀기 위해 저는 크게 두 가지 단계를 설계했습니다.

1. **Bradley-Terry Prior + Empirical 하이브리드 행렬**
   배틀로얄 특유의 비추이적인 상성("가위바위보"같이 A가 B를 이기고, B가 C를 이기지만 C가 A를 이기는 현상)을 살리기 위해 실제 관측된 승률(Empirical Matrix)과, 만난 적 없는 유저들을 위해 Bradley-Terry 확률을 하이브리드로 결합했습니다. 시간이 지남에 따라 실력이 변하는 '메타 드리프트'를 반영하기 위해 과거 데이터일수록 페널티를 주는 **지수 감쇠(Exponential Time Decay)**를 적용했습니다.

2. **Markov Chain을 이용한 Expected Kills (기댓값) 분석**
   흔히 사용하는 몬테카를로 시뮬레이션 대신, 마코프 체인을 활용해 5명의 건틀릿 매치 순서에 따른 생존 확률과 킬 기댓값을 결정론적(Deterministic)으로 계산했습니다. 이 방식을 통해 매치 당 정확히 4킬이 유지되는 성질을 완벽하게 보존하여 총합 5800킬이라는 제약 조건을 오차 없이 100% 만족시켰습니다.

## 2. 뼈아픈 실수 (What Went Wrong)

알고리즘의 수학적 증명은 완벽했지만, 모델의 성능을 평가하고 파라미터를 찾는 **검증 파이프라인(Validation Pipeline)**에 치명적인 개념적 결함이 숨어 있었습니다.

### 🚩 실수 1. 시계열 예측 범위의 미스매치 (Time-Horizon Mismatch)
훈련 후 하이퍼파라미터(`lambda`, `alpha`)를 찾기 위해 Grid Search를 돌릴 때, **"Day 1~20을 학습시키고, 바로 다음 날인 Day 21을 얼마나 잘 맞추는지(1-Step Ahead)"**만을 기준으로 삼았습니다.
바로 다음 날을 예측할 때는 당연히 직전 날의 실력이 가장 중요하므로, Grid Search는 과거 15일 치 데이터를 거의 다 버리고 최근 5일에 가중치를 몰아주는 **극적으로 강한 시간 감쇠(`lambda=8.0`)**를 최적의 정답으로 뱉어냈습니다.

하지만 우리가 실제로 맞혀야 하는 본 게임은 **Day 22부터 Day 50까지, 무려 한 달(29일) 구간의 평균적인 킬 수**였습니다.
하루 뒤를 맞추기 위해 과적합된 파라미터(어제오늘 폼이 좋은 플레이어에게 가중치를 극단적으로 몰아준 스냅샷)를 무려 30일 미래까지 그대로 연장해버리니, 장기적인 평균 회귀(Mean-Reversion)를 전혀 잡아내지 못하고 오차가 폭발해 버린 것입니다.

### 🚩 실수 2. 무너진 L2 Regularization 스케일 조정
과거 데이터를 강하게 버리는 `lambda=8.0`을 채택하면서 학습에 기여하는 그래디언트(Gradient)의 총 질량(유효 샘플 사이즈)이 4200개에서 525개 수준으로 크게 쪼그라들었습니다. 
하지만 이를 반영하지 않고 L2 정규화 페널티를 고정 상수(`1.0`)로 넣어버려, 그래디언트 총량 대비 패널티가 기형적으로 크게 작용하며 플레이어들의 실력(Skill) 값이 비정상적으로 억눌렸습니다.

## 3. 어떻게 고쳤는가? (Resolution)

대회 종료 후, 문제의 근본 원인을 제거하기 위해 코드 3가지를 리팩터링했습니다.

### 🛠️ 1. 장기 보폭 검증 (Long-Horizon Validation) 도입
`grid_search.py`가 단순히 내일(Day 21)을 벤치마킹하는 것을 멈추고, 장기적인 추세를 잡을 수 있도록 검증 단위를 크게 늘렸습니다.
* **학습 구간:** 전반부 (Day 1 ~ 11)
* **테스트 구간:** 후반부 (Day 12 ~ 21 전체 10일)
이처럼 10일간의 평균을 잘 맞추도록 밸리데이션 타겟을 조정하여 모델이 장기 미래를 예측하는 훈련을 하도록 검증 파이프라인을 완전히 갈아엎었습니다.

### 🛠️ 2. 유동적인 L2 페널티 스케일링 체계 도입
시간 감쇠(`lambda`)가 변함에 따라 그래디언트의 질량이 바뀌더라도 정규화가 일관되게 적용될 수 있도록, 가중치 총합 비율을 계산해 페널티 상수에 곱해주었습니다.
```python
# 수정 전
skills[i] += lr * (grads[i] - 1.0 * skills[i])

# 수정 후: 감쇠율(lambda)에 비례하는 유효 질량(sum_weights / N)을 적용
sum_weights = sum(math.exp(-lambda_val * (N - 1 - idx) / N) for idx in range(N))
l2_lambda = 0.5 * (sum_weights / N) 
skills[i] += lr * (grads[i] - l2_lambda * skills[i])
```

### 🛠️ 3. 최적 파라미터 재탐색 및 하이브리드 비율(Alpha) 밸런싱
수정된 장기 셋업으로 그리드 서치를 돌린 결과, 모델이 드디어 단기 노이즈에 집착하지 않고 전체적인 평균과 장기적인 실력 편차를 잡아내는 유의미한 예측값을 도출하게 되었습니다.

## 4. 회고 및 깨달음 (Takeaway)

이번 해커톤을 통해 뼈저리게 배운 점은 **"알고리즘의 수학적 증명이 아무리 완벽하더라도, 모델을 평가하는 Validation Task가 실제 Test Task의 시간적 스케일(Time-Horizon)과 다르면 모델은 반드시 잘못된 방향으로 최적화된다"**는 것입니다.

모델 구조를 짤 때 쏟았던 노력의 절반만이라도 데이터를 분할하고 평가하는 방법(Evaluation Methodology)을 설계하는 데 할애했다면 점수를 크게 방어할 수 있었을 것입니다. 너무나 아쉬운 결과지만, 이번 딥 다이브를 통해 시계열 검증 설계의 중요성을 평생 잊지 않게 될 훌륭한 오답 노트를 얻었습니다.
