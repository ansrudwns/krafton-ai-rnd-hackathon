# [KRAFTON AI 해커톤 회고] 절반의 성공: 완벽했던 맵리듀스(Map-Reduce)가 타임어택 중 멈춰버린 이유

해커톤 Day 2 결선 과제는 극한의 타임어택(Time-Attack)이었습니다. **최대 20분 길이의 영상 20개를 수령 후 단 15분 만에 모두 분석해 객관식 정답을 제출해야 하는 극악의 난이도**였죠. 

이를 해결하기 위해 우리 팀은 시공간 압축(FFmpeg)과 ‘이기종 맵리듀스(Heterogeneous Map-Reduce)’ 아키텍처를 결합해, 영상을 잘게 쪼개어 가벼운 모델(Flash)로 1차 스캔을 한 뒤 무거운 모델(Pro)로 모아서 추론하는 파이프라인(`VideoAgent V3`)을 짰습니다. 다중 구글 계정 API 키를 로드밸런싱하고 동시성 제어(Semaphore)까지 완벽하게 스케줄링했지만... **안타깝게도 시스템은 20개 중 절반 정도를 처리하다가 완전히 멈춰(Hang)버렸고, 결국 미완성 상태로 돌릴 수밖에 없었습니다.**

대회가 끝난 후 깊은 사후 분석(Post-mortem)을 통해 **코드 어디에서 데드락(Deadlock)이 걸렸는지** 파악했습니다. 분산 처리와 비동기(Async) 파이프라인을 설계할 때 누구나 한 번쯤 겪을 수 있는 뼈아픈 함정들을 공유합니다.

---

## 1. 우리가 구성했던 혁신적인 파이프라인 (What We Built)

1. **초고속 시공간 다운샘플링**: 무거운 영상을そのまま 클라우드에 올리면 대역폭이 터집니다. 그래서 `FFmpeg`을 이용해 화질을 낮추고 초당 1프레임(1fps)으로 강제 압축해, 500MB 영상을 단 5초 만에 4MB로 압축하는 극강의 전처리를 구성했습니다.
2. **Query-Aware Map-Reduce Tracking**: 압축된 영상을 작게(예: 5분 단위) 분할(Chunking)하여, 각 청크를 초고속 모델(`Gemini Flash`)이 동시에 시청하게 했습니다. 그리고 질문에 해당하는 핵심 객체의 타임라인만 텍스트 로그로 추출해 냈습니다.
3. **최종 오케스트레이션**: 추출된 텍스트 로그들을 논리 추론 최강자(`Gemini Pro`)에게 한 번에 넘겨 객관식 정답만을 뽑아내도록 구성했습니다. 
4. **API 트래픽 한계 방어망**: 429 에러(Rate Limit)를 피하기 위해 N개의 API 키를 Round-Robin으로 순회하고, `Semaphore`를 이용해 통신 채널 수를 완벽하게 잠그는 글로벌 락 제어를 적용했습니다.

로직상으로는 구멍이 없는, 정말 방탄(Bulletproof)과 같은 아키텍처였습니다. **그렇다면 왜 멈췄을까요?**

## 2. 시스템이 멈춰버린 3가지 치명적 결함 (What Went Wrong)

문제는 인공지능 모델 퀄리티가 성능이 아니라 **비동기 프로그래밍(Asyncio)의 예외 처리와 클라우드 API 자체의 엣지 케이스(Edge Case)**에 있었습니다.

### 🚩 치명적 함정 1. "PROCESSING" 무한 대기 (업로드 타임아웃의 부재)
동영상을 청크(Chunk)로 잘라서 업로드할 때, Gemini API는 서버에서 백그라운드 처리를 거치며 상태를 `PROCESSING`에서 `ACTIVE`로 넘겨줍니다. 여기서 작성한 코드는 다음과 같습니다.
```python
while uploaded_file.state.name == "PROCESSING":
    time.sleep(2) 
    uploaded_file = client.files.get(name=uploaded_file.name)
```
**문제:** 영상 분할 과정에서 메타데이터가 미세하게 손상된 청크가 업로드될 경우, API 서버는 에러를 뱉지 않고 **영구적으로 `PROCESSING` 상태에 빠지는 버그**가 간혹 발생합니다.
타임아웃(Timeout) 코드를 짜두지 않았기 때문에, 단 1개의 파일이라도 이 상태에 빠지면 해당 쓰레드(Thread)는 영원히 `time.sleep(2)`을 반복하며 **할당된 세마포어(Semaphore) 슬롯을 영영 반납하지 않는 데드락**을 유발했습니다.

### 🚩 치명적 함정 2. `asyncio.gather`의 예외 전파와 '좀비 태스크(Zombie Task)' 메모리 누수
영상 1개가 4개의 청크로 나뉘어 동시에 `process_chunk` 태스크로 묶여 병렬 실행되었습니다.
이때 만약 1번 청크가 API 에러로 `Exception`을 뱉는다면?
```python
# 여러 청크를 동시에 실행
results = await asyncio.gather(*chunk_tasks)
```
파이썬의 `asyncio.gather`는 에러가 발생하면 즉시 예외를 부모 함수로 던지고 취소(abort)됩니다. **하지만 백그라운드에 남아있던 2, 3, 4번 청크의 Task들은 종료되지 않고 '좀비' 상태로 계속 실행됩니다!** 
에러를 뱉은 비디오는 프로세스를 'X'로 처리하고 넘어갔지만, 좀비 태스크들은 뒷단에서 계속 API 키를 잡아먹고 세마포어를 점유하게 됩니다. 결국 이런 에러가 몇 번 누적되자 글로벌 세마포어(제한된 동시성 슬롯)가 좀비들로 100% 가득 차면서 전체 시스템 작동이 **병목 정지**했습니다.

### 🚩 치명적 함정 3. 재시도(Retry) 로직의 키 바인딩 에러
API 한도를 초과했을 때(429 에러), 시스템은 15초를 대기했다가 무려 4번이나 끈질기게 재시도하는 훌륭한 생존 로직을 넣었습니다.
```python
client = self.get_client() # 함수 시작 시 키를 1개 할당 받음
for inference_attempt in range(4):
    try:
        # API 호출!
    except:
        if '429' in err: time.sleep(15) # 한도 초과 시 15초 대기
```
하지만 **이미 한도를 초과해 뻗어버린 API Key(`client`)를 버리지 않고, 그대로 잡은 상태로 15초를 대기했다가 "같은 키"로 재시도**를 했습니다.
가용 API 키 목록이 많이 남아있음에도 불구하고, 신선한 키를 꺼내오지 않고 소진된 키 하나를 붙잡고 15초씩 낮잠을 잤기 때문에, 병렬 처리 속도의 장점이 полностью 파괴되는 지연이 발생했습니다.

## 3. 완벽한 백엔드 아키텍처를 위한 보완책 (How to Fix It)

대회는 끝났지만, 이를 프로덕션 레벨의 완벽한 아키텍처로 바꾸려면 단 3줄의 전략만 변경하면 됩니다.

1. **API Polling 락에 Hard Timeout 걸기**
   * 영원한 `PROCESSING`을 막기 위해 60초가 넘어가면 구성을 강제로 끊고(Timeout Exception) Fallback으로 빠지는 예외 처리를 추가해야 합니다.
2. **비동기 태스크 클린업 (`return_exceptions=True`)**
   * `asyncio.gather(*chunk_tasks, return_exceptions=True)` 옵션을 주면, 하나의 청크가 죽어도 다른 청크들이 세마포어를 완전히 반환할 때까지 통제권을 잃지 않고 조화롭게 종료됩니다. (혹은 강제로 캔슬하는 관리자 노드가 필요합니다).
3. **루프 안으로 `get_client()` 밀어 넣기**
   * 에러가 발생한 즉시 "이 API 키는 망가졌어!"라고 판단하고, 다음 바퀴 재생 시 `client = self.get_client()`를 루프 **내부**에서 호출하여 새로운 구글 계정 키로 즉각 스왑(Dynamic Key Rotation)해야만 속도 저하를 막을 수 있습니다.

## 4. 맺음말
로컬 테스트에서는 영상 1~2개로 완벽하게 돌아갔고, 아키텍처 구상과 아이디어(초저화질 분할, 맵리듀스)도 정말 좋았습니다. 하지만 실전 대회장에서 파이프라인에 대량 폭격을 가했을 때, 네트워크의 엣지 케이스는 가차 없이 찾아옵니다.

**분산 처리의 세계에서는 인공지능 모델 튜닝만큼이나 "시스템이 언제든 예기치 못하게 죽을 수 있다는 방어적 프로그래밍(Defensive Programming)"이 필수적임**을 아주 값비싸고 확실하게 배운, 후회 없는 무대였습니다.
